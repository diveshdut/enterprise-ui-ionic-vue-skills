# Figma Enterprise Workflow Index

## Source Of Truth

Use the live Figma page as the primary navigation layer:
[Enterprise Workflow Index](https://www.figma.com/design/1wantteXXUlzPRYev1X5gr/Ionic-8-Material-UI-Kit--Community---Community-?t=hEXHxO7yI4T4m4pV-1&node-id=34926-1223).

Use the broader Ionic 8 Material UI Kit file as the visual source of truth:
[Ionic 8 Material UI Kit Community](https://www.figma.com/design/1wantteXXUlzPRYev1X5gr/Ionic-8-Material-UI-Kit--Community---Community-?node-id=12167-10&t=hEXHxO7yI4T4m4pV-1).

This index helps agents and contributors choose the right foundation,
workflow reference, UX pattern, and Ionic component set without manually
navigating every page and layer first.

If you need to move from design frames to implementation code, see
[design-code-mapping.md](design-code-mapping.md).

## Preservation Rule

All existing pages, canvases, sections, frames, layers, components, styles,
variables, and workflow examples in this Figma file are reference sources. Do
not edit, move, rename, delete, restyle, or overwrite them while creating a new
design unless the user explicitly authorizes changes to identified existing
content. Add new design work in a new page or a clearly designated new area by
default.

## How To Use This Index

1. Start with the live
   [Enterprise Workflow Index](https://www.figma.com/design/1wantteXXUlzPRYev1X5gr/Ionic-8-Material-UI-Kit--Community---Community-?t=hEXHxO7yI4T4m4pV-1&node-id=34926-1223).
2. Check foundations when selecting controls, iconography, or semantic colors.
3. Select the closest workflow category before inspecting individual screens.
4. Use the selection taxonomy when the requirement describes a screen type
   rather than naming a product workflow.
5. Use pattern cards and the decision layer to map UX patterns to Ionic
   components, states, and responsive behavior.

### State Coverage Checklist

Before a design is implementation-ready, check state coverage for each major
workflow view.

| State Group | Required States |
| --- | --- |
| Data states | Loading, empty, no results, partial data, stale data. |
| Action states | Saving, success, validation error, destructive confirm, undo path. |
| Access and risk | Permission denied, read-only mode, offline/sync pending, conflict detected, audit required. |
| Responsive states | Mobile drill-down, desktop dense list, tablet split view, keyboard/scanner input, long text wrapping. |

### Component Authenticity Guidance

Use the live Component Authenticity Guidance area in the
[Workflow Design Decision Layer](https://www.figma.com/design/1wantteXXUlzPRYev1X5gr/Ionic-8-Material-UI-Kit--Community---Community-?node-id=34935-1223&t=hEXHxO7yI4T4m4pV-1)
before creating or revising enterprise screens.

Checklist:

- Visible controls must be real Ionic instances or valid component slots.
- Component-owned labels stay inside the component layer tree.
- A hidden Ionic instance plus custom visible overlay does not count as reuse.
- Audit visible primitives outside instances before delivery.
- Direct text is only for standalone titles, counts, notes, and guidance.

Element rules:

- Searchbar: use the Searchbar instance with text inside component content.
- Filters and sort: use compact Chips or small outline controls.
- Buttons: use the Button instance label layer.
- Cards: use the Card surface/content pattern.
- Items and results: use Item start/end slots.
- Inputs and selects: use Input/Select labels and values.
- Status: use Badge or Chip with semantic color.
- Toolbar: use Toolbar slots and Ionicons.
- Direct text: never use direct labels for component-owned text.

Before saying done, inspect screenshots, check visible component instance
counts, list legitimate direct text layers, verify hidden instances are only
internal inactive variants, and name the inspected workflow references and
component foundations.

### Workflow Anatomy

Most enterprise workflows can be decomposed into this sequence:

`Entry -> Search -> Results -> Detail -> Action -> Confirm -> Status`

Use the anatomy to verify that a design has clear entry points, discoverable
records, an inspectable detail view, a deliberate action model, confirmation
where risk exists, and visible status/feedback after work is performed.

### Decision Rules

- Use a modal for focused interruption, confirmation, or short contextual
  editing. Avoid long multi-step record creation inside a modal.
- Use tabs when detail sections are peers and users need frequent switching.
- Use accordions when sections are optional, review-oriented, or progressively
  disclosed.
- Use mobile drill-down instead of dense split views on narrow screens.
- Use chips and badges for compact metadata; use semantic color only when it
  communicates state, severity, or selection.
- Use progress indicators for long work, queued jobs, receiving/completion
  status, and import/export processing.
